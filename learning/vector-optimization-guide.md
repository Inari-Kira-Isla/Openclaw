## 🎯 核心理念

> 「有用」的向量資料 = 精煉的語義塊 + 明確的元數據標籤 + 強制的上下文注入

---

## 🏗️ 四個維度優化方案

### 1. 語義分塊的「原子化」與「上下文注入」

| 問題 | 解決方案 |
|------|----------|
| Chunk 太長，語義稀釋 | 每個標題只講一件核心事情 |
| Chunk 太短，失去上下文 | 手動加入 Context Breadcrumb |

**範例**：
```
[領域：海膽鑑定 / 豐洲市場 / 成本分析]
這批 A 級馬糞海膽的到岸價格...
```

---

### 2. 建立「Metadata（元數據）」影子索引

| 元數據 | 說明 |
|--------|------|
| 角色歸屬 | Athena（行政/SOP）、QI（分析）、DDH（決策）|
| 時間戳 | 2026年與2025年權重不同 |
| 關聯節點 | 對應到哪個 .md 檔案的哪一行 |

**應用**：
- 檢索時設定過濾：「僅檢索 2026 年且標記為『市場分析』的向量」

---

### 3. 預處理：摘要式向量記錄

**操作流程**：
1. 寫下原始筆記
2. 小龍蝦/LLM 生成 200 字內精煉摘要
3. 向量化這個摘要
4. 連結到原始長文

**優點**：
- 摘要語義密度最高
- AI 透過摘要命中的機率更高

---

### 4. 建立「定義集」

| 用途 | 說明 |
|------|------|
| 專門詞彙定義 | 「landed cost」、「小龍蝦框架」、「手牌」等 |
| 增強聚類效應 | 高頻率引用讓詞彙在向量空間中更精準 |

## 🛠️ 行動建議

### 建立 Memory_Index 資料夾

在 Obsidian 根目錄建立 `Memory_Index/`

### memorySearch 規則

| 規則 | 說明 |
|------|------|
| Index Sync | 每次偵測到檔案變動，僅針對變動塊進行增量 Embedding |
| Vector Store | 使用 ChromaDB 或 LanceDB，生成 .bin/.index 檔案可直接放入雲端同步 |
