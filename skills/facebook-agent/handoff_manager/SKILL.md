# Handoff Manager — 轉接管理

## 功能說明

管理 Facebook Messenger 對話從 AI 轉接到人工客服的完整流程，確保客戶問題不遺漏、交接資訊完整、跟進有追蹤。

## 轉接觸發條件

### 自動轉接

| 條件 | 優先級 | 說明 |
|------|--------|------|
| 投訴/負面情緒 | P0 | 偵測到抱怨、生氣、不滿 |
| 明確要求人工 | P0 | 「轉人工」「我要跟真人講」「找客服」 |
| 訂單/財務問題 | P1 | 需要系統操作權限 |
| AI 回覆 3 次仍未解決 | P1 | 超出 AI 處理能力 |
| 法律相關問題 | P1 | 風險議題，不可由 AI 回覆 |

### 手動轉接
使用者或管理員可隨時手動觸發轉接。

## 操作步驟

1. 偵測到轉接條件（來自 question_classifier 或自動判斷）
2. 生成交接摘要：用戶資訊 + 對話重點 + 轉接原因
3. 透過 Telegram 通知值班管理員
4. 在本地資料庫（SQLite）建立待辦事項
5. 向客戶發送轉接提示訊息
6. 標記對話狀態為「已轉接」
7. 啟動跟進計時

## 交接摘要格式

```
轉接交接單
用戶：[姓名]（FB ID: [ID]）
時間：[時間]
對話摘要：[重點內容]
轉接原因：[原因]
優先級：高 / 中 / 低
待處理：聯繫用戶 → 解決問題 → 回覆結果
```

## 跟進機制

| 時間 | 動作 |
|------|------|
| 轉接後 30 分鐘 | 若未處理，再次提醒值班人員 |
| 轉接後 1 小時 | 若仍未處理，升級通知管理者 |
| 處理完成 | 標記「已處理」，記錄解決方案，關閉待辦 |

## 工具指引

- **Telegram Bot**：通知值班管理員
- **SQLite（cs_customers.db）**：建立待辦事項、記錄交接單（透過 cs_customer_db.py）
- **conversation_logger**：取得對話歷史供交接
- **提醒系統**：跟進計時與升級通知

## 錯誤處理

| 情境 | 處理方式 |
|------|----------|
| Telegram 通知失敗 | 改用 Email 通知，並重試 Telegram |
| 無值班人員 | 告知客戶預計回覆時間，記錄至本地資料庫待辦 |
| 資料庫寫入失敗 | 將交接資訊暫存於對話中，稍後重試 |
| 客戶在轉接等待中離開 | 保留交接單，待值班人員主動聯繫 |

## 使用範例

- 客戶說「我要跟真人講！」→ 立即轉接 + 通知
- AI 回覆 3 次客戶仍不滿意 → 自動轉接 + 摘要
- 客戶詢問退款流程 → 標記訂單類 → 轉人工處理

## 防護規則

- 轉接時不可洩露內部處理流程給客戶
- P0 轉接不可延遲，必須立即執行
- 交接摘要需客觀中立，不可加入 AI 的主觀判斷
- 未經處理的轉接單不可自動關閉
- 所有轉接記錄須保留至少 90 天
