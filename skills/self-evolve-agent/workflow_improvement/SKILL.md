# 工作流改進

分析現有工作流的瓶頸與問題，設計優化方案並追蹤改進效果。

## 操作 / 工作流程

1. **工作流盤點** — 讀取目標工作流的步驟與配置
2. **瓶頸分析** — LLM 從三個維度評估：
   - 效能瓶頸：耗時步驟、等待時間、資源衝突
   - 錯誤模式：失敗步驟、重試次數、錯誤類型
   - 複雜度：步驟數量、依賴深度、分支數量
3. **優化方案設計** — 依分析結果提出改進策略：
   - 並行化：將可同時執行的步驟並行處理
   - 簡化：合併冗餘步驟，減少依賴鏈
   - 快取：對重複計算加入快取機制
   - 重試優化：智慧重試 + 快速失敗 + 降級處理
4. **效果預估** — 估算優化後的改善幅度（速度、可靠性、複雜度）
5. **記錄追蹤** — 將方案存入 `memory_search`，設定追蹤週期：
   - 發布後 24 小時、1 週、1 個月

## 參數

| 參數 | 類型 | 預設 | 說明 |
|------|------|------|------|
| workflow | string | 必填 | 目標工作流名稱或描述 |
| focus | string | "all" | 分析焦點：performance / errors / complexity / all |
| action | string | "analyze" | 動作：analyze / implement / track |

## 輸出格式

```
🔄 工作流改進報告 — {workflow}

現況分析：
- 總步驟數：{steps}
- 平均執行時間：{avg_time}
- 錯誤率：{error_rate}%

瓶頸：
{foreach bottleneck}
  - {step}：{issue}（影響：{impact}）
{/foreach}

優化方案：
{proposal}

預估改善：
- 速度：{speed_improvement}
- 可靠性：{reliability_improvement}
```

## 錯誤處理

| 錯誤 | 處理 |
|------|------|
| 工作流不存在 | 請用戶描述工作流程，從描述中分析 |
| 無歷史數據 | 僅做靜態分析（結構複雜度），建議累積數據後再分析 |
| 優化方案互相衝突 | 排序後建議逐步實施，避免同時改動 |
| 追蹤指標無改善 | 重新分析，調整方案或回退 |

## 使用範例

- "記憶攝入的流程能不能更快"
- "分析一下 BNI 提醒工作流的瓶頸"
- "最近哪個工作流錯誤最多"
