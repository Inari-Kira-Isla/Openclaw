# 負載平衡

監控各 Agent 與服務的負載狀況，動態分配任務，確保系統資源均衡使用。

## 操作 / 工作流程
1. 用 `browser` 收集各節點/Agent 的負載指標：
   - CPU 使用率、記憶體使用率
   - 任務佇列長度、回應時間
2. 評估各節點健康狀態（健康 / 警告 / 過載 / 離線）
3. 根據負載狀況選擇分配策略：
   - 均衡模式：輪流分配（Round Robin）
   - 最低負載優先：分配給負載最低的節點
   - 加權分配：依節點能力加權
4. 若偵測到節點過載或離線，觸發故障轉移
5. 用 `message` 通知 Telegram 負載異常或故障轉移事件
6. 將負載數據存入記憶，追蹤長期趨勢

## 參數
| 參數 | 類型 | 預設 | 說明 |
|------|------|------|------|
| strategy | string | least_load | 分配策略：`round_robin` / `least_load` / `weighted` |
| check_only | bool | false | 僅檢查不重新分配 |

## 輸出格式
```
⚖️ 負載平衡報告

| 節點 | CPU | 記憶體 | 佇列 | 狀態 |
|------|-----|--------|------|------|
| {node} | {cpu}% | {mem}% | {queue} | {status} |

當前策略：{strategy}
整體負載：{overall_load}%

⚠️ 異常：
- {node}: {issue}
- 處理：{action_taken}
```

## 錯誤處理
| 錯誤 | 處理 |
|------|------|
| 節點無回應 | 標記為離線，將其任務轉移至其他節點 |
| 所有節點高負載 | 發出緊急警報，建議暫緩非關鍵任務 |
| 指標收集逾時 | 使用最近一次的快取數據，標注數據時效 |

## 使用範例
- "目前各節點的負載狀況"
- "系統有過載嗎"
- "重新分配一下任務負載"
