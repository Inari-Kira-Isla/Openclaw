# 失敗分析

分析系統或任務失敗的根本原因，提出修復建議與預防措施。

## 操作 / 工作流程

1. **收集錯誤資訊** — 取得錯誤訊息、時間戳、上下文（用戶輸入、系統狀態）
2. **錯誤分類** — 依層級分類：
   | 層級 | 類型 | 範例 |
   |------|------|------|
   | L1 | 輸入錯誤 | 參數格式錯誤、缺少參數 |
   | L2 | 執行錯誤 | API 呼叫失敗、超時 |
   | L3 | 邏輯錯誤 | 計算錯誤、判斷失誤 |
   | L4 | 系統錯誤 | 記憶體不足、磁碟滿 |
3. **根因分析** — 使用 5 Whys 技術逐層追溯根本原因
4. **修復建議** — 提出具體修復方案，標註優先級與工作量
5. **預防措施** — 建議長期改善項目（驗證、錯誤處理、監控、測試）
6. **記錄教訓** — 使用 `memory_search` 查詢是否為已知問題，新問題寫入記憶

## 參數

| 參數 | 類型 | 預設 | 說明 |
|------|------|------|------|
| error_message | string | — | 錯誤訊息 |
| context | object | {} | 錯誤發生時的上下文資訊 |
| severity | string | auto | 嚴重程度：critical / high / medium / low / auto |
| include_prevention | boolean | true | 是否包含預防措施建議 |

## 輸出格式

```
🔍 失敗分析報告
錯誤類型：[L2 執行錯誤]
根本原因：[具體原因描述]

🔧 修復建議：
1. [修復方案] — 優先級：[high] / 工作量：[小]

🛡️ 預防措施：
1. [長期改善項目]
```

## 錯誤處理

| 錯誤 | 處理 |
|------|------|
| 錯誤資訊不足 | 列出需要的額外資訊，請求補充 |
| 無法判定根因 | 列出可能原因，建議逐一排查 |
| 已知問題重複出現 | 標記為重複，提高修復優先級 |
| 分析過程本身失敗 | 回傳原始錯誤資訊，建議人工排查 |

## 使用範例

- "這個任務為什麼失敗了？"
- "分析一下這個錯誤的根本原因"
- "最近常出現這個錯誤，幫我查"
