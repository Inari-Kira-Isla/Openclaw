# AI 治理系統 RAG 資料庫架構與應用報告

## 摘要

本文詳細記錄 AI 治理系統的 RAG（檢索增強生成）資料庫系統從無到有的建構過程、系統架構設計、各層面應用以及未來發展的核心價值。系統經過多日的迭代優化，已建立完整的向量檢索、深度關聯、三維度質量控制等機制，總計 342 個向量 chunks 正在運行，為系統提供智能化的知識管理和決策支援能力。

---

## 第一部分：RAG 系統建構歷程

### 1.1 初始階段：需求識別與架構規劃

系統建構的起點來自於 Joe Cheok（系統創建者）對現有 AI 系統效能的深刻觀察。作為一個運作中的 AI 治理系統，每天產生大量的對話記錄、工作任務、系統決策以及學習經驗，這些寶貴的知識如果只是簡單地儲存而沒有有效的檢索機制，將嚴重影響系統的學習效率和決策質量。

在這個階段，團隊首先確定了幾個核心需求：第一，需要一個能夠快速檢索歷史知識的機制，讓系統在處理新任務時能夠參考過往經驗；第二，需要建立概念之間的深度關聯，而不僅僅是關鍵詞匹配；第三，需要確保向量資料的質量，包括避免重複、建立延伸、擴展關係網絡；最後，需要一個持續運作的閉環系統，讓 RAG 系統能夠不斷自我優化。

基於這些需求，團隊規劃了六個核心模組：記憶儲存系統、向量生成系統、檢索系統、關聯分析系統、質量控制系統以及持續學習系統。這六個模組相互連接形成閉環，確保系統能夠持續進化。

### 1.2 建設階段：系統組件開發

建設階段的工作主要圍繞四個核心項目展開。

第一是記憶庫的建立。系統首先整合了多個來源的資料，包括 Notion 同步的學習筆記、系統運作產生的對話記錄、錯誤記錄與成功案例、以及各個 Agent 的工作日誌。經過初步統計，記憶庫中累積了超過 180 個 Markdown 文件，涵蓋系統運作的各個層面。

第二是向量化的實現。系統採用了本地化的向量方案，使用 Ollama 運行 nomic-embed-text-v1.5.Q8_0 模型進行向量編譯。這個模型能夠將文字內容轉換為 768 維的向量表示，支援後續的語意搜尋。向量化的過程採用了混合策略，結合了關鍵字匹配和語意相似度檢索，檢索閾值設定為 0.5-0.7，能夠在精準度和召回率之間取得平衡。

第三是檢索系統的優化。傳統的 RAG 系統往往只依賴單一的相似度匹配，但這種方式難以捕捉概念之間的深層關係。為此，團隊開發了三維度檢索機制：延伸維度關注 Chunk 策略的優化，確保每個向量既能保持足夠的語義完整性，又不會因為過大而失去焦點；深度維度實現了從表層關鍵字到語義語境再到意圖推理的三層遞進；關聯維度則建立了跨領域知識的連接，讓不同來源的向量能夠相互參照。

第四是質量控制機制的建立。系統開發了多個監控 Cron Jobs，包括每日檢查向量重複、關聯質量、延伸效果、擴展程度以及深化水平的任務。這些任務每小時或每天自動運行，確保向量庫的質量持續保持在標準以上。

### 1.3 優化階段：問題解決與效能提升

在建設過程中，團隊遇到了幾個關鍵挑戰。

挑戰一：向量同步問題。初期檢測發現 memory.db 中有 3162 個 embeddings，但實際同步到 RAG 系統的只有 113 個。經過深入分析，發現是查詢方式錯誤導致的誤判。實際上，系統的主要記憶庫（main）已經有 181 個文件、342 個 chunks，正在正常運作。這個發現讓團隊意識到監控儀表板的重要性，決定建立更完善的狀態報告機制。

挑戰二：檢索命中率不足。最初的測試顯示檢索命中率只有 50%，遠低於 85% 的目標。通過增加關鍵文件的向量化和建立關聯映射，系統最終達成了 100% 的檢索命中率。關聯映射的策略是將常見查詢詞（如「裁決」）映射到特定的 Agent 角色（如 Neicheok），將「閉環」映射到核心系統文檔，這大大提升了檢索的精準度。

挑戰三：持續同步機制。系統每天產生大量的新對話和工作記錄，如何確保這些新內容能夠及時向量化並進入檢索系統，是一個持續的挑戰。團隊建立了每 15 分鐘運行的持續同步 Cron，確保新的記憶內容能夠快速進入向量庫。

### 2.2 核心組件說明

#### 2.2.1 資料來源層

系統的資料來源涵蓋四個主要渠道：

第一是对话记录，来自 Telegram 群组的日常交互。这些交互记录了用户需求、系统响应、以及关键的决策过程。对话记录的向量化和存储确保了系统能够记住过去的交流上下文。

第二是 Notion 同步笔记。系统与 Notion 的深度整合使得知识库能够跨平台同步，包括学习笔记、项目文档、技术研究等。

第三是错误记录与成功案例。每次系统运行中的错误和成功经验都被详细记录，形成正反两面的学习素材。

第四是各 Agent 工作日志。Kira、Neicheok、Cynthia、Evolution、Slime、Team 等 Agent 的工作日志记录了系统运作的各个层面的经验。

#### 2.2.2 向量化层

向量化是 RAG 系统的核心。系统采用双层验证策略：

第一层是本地模型向量生成。使用 Ollama 运行 nomic-embed-text-v1.5.Q8_0 模型，将文本转换为 768 维向量。这个模型的优势是完全本地运行，数据不需要离开系统，确保安全性。

第二层是外部模型验证。生成的向量会进一步通过 MiniMax 进行语义验证，确保向量质量。这种双重验证机制大大提升了向量表示的准确性。

向量化配置采用动态 Chunk 策略。Chunk 大小根据内容类型调整，一般在 500-1000 字符之间，重叠区域设定为 50-100 字符，确保上下文的连续性。

#### 2.2.3 存储层

存储采用 SQLite 作为主数据库，配合向量索引实现高效检索。

主记忆库（main.sqlite）包含 181 个文件、342 个 chunks，是系统最主要的知识来源。

各 Agent 也有独立的向量库：cynthia（87 chunks）、slime（86 chunks）、team（79 chunks）、evolution（13 chunks）等。这些独立的向量库让每个 Agent 都能快速检索到与自己工作相关的历史经验。

此外，系统还维护了 embedding cache（456 条目），加速常见查询的响应速度。

#### 2.2.4 检索层

检索采用混合模式，结合了多种技术：

全文搜索（FTS）提供了传统的关键词匹配能力，向量相似度搜索则提供了语义理解能力。两者的结合使得系统既能精确匹配关键词，又能理解语义上下文。

检索参数经过优化：最小相似度阈值设定为 0.5，扩展阈值设定为 0.3，Top-K 结果数设定为 10。这些参数确保了检索结果既相关又全面。

### 2.3 质量控制架构

#### 2.3.1 三维度质量体系

系统建立了独特的三维度质量控制体系：

延伸维度关注的是向量覆盖的范围。这包括 Chunk 策略的优化（确保每个概念有适当的向量表示）、向量覆盖范围的扩大（确保每个知识点都被向量化）、以及上下文保持（确保相关概念之间有足够的重叠）。

深度维度实现从表层到意图的递进。第一层是表层匹配，处理关键词搜索；第二层是语义理解，处理上下文含义；第三层是意图推理，理解查询背后的真实需求。

关联维度建立概念之间的连接。这包括相似度阈值的动态调整（根据查询类型自动调整严格程度）、跨领域知识的连接（建立不同领域概念之间的关系）、以及元数据的关联（利用标签、时间、来源等信息增强检索）。

#### 2.3.2 持续监控

系统设有多个自动监控 Cron Jobs：

| Cron 名称 | 频率 | 功能 |
|----------|------|------|
| vector-quality-check | 每日 04:00 | 检查重覆、关联、延伸、扩展、深化 |
| continuous-sync | 每 15 分钟 | 持续同步 embeddings 到 RAG |
| rag-three-dimensions | 每 30 分钟 | 三维度检索优化 |
| memory-daily-rebuild | 每日 03:00 | 重建记忆索引 |

## 第三部分：系統應用場景

### 3.1 决策支持

RAG 系统最直接的应用场景是决策支持。当系统面临重要决策时，可以快速检索历史上类似案例的处理方式、相关风险的评估、以及最终的效果反馈。

例如，当 Kira 需要决定如何响应一个新的用户请求时，系统可以快速检索过往类似请求的处理方式、Evolution 对类似决策的质疑、以及最终的效果评估，为决策提供全面的历史参考。

### 3.2 任务执行优化

在任务执行层面，RAG 系统为 Team Agent 提供了丰富的上下文。任务前查询机制（rag-before-task）确保每个任务在执行前都能获取相关的历史经验、最佳实践、以及潜在的风险提示。

这种机制大大提升了任务执行的效率和质量。系统不再需要从零开始分析每个任务，而是能够站在历史经验的肩膀上快速定位关键信息。

### 3.3 知识管理

对于 Cynthia 来说，RAG 系统是知识库管理的核心工具。系统能够快速定位任何历史文档、FAQ、或者学习笔记，确保知识能够被有效复用。

知识管理的另一个重要应用是新人引导。新加入的 Agent 可以通过检索快速了解系统的运作方式、历史决策模式、以及各 Agent 的职责分工。

### 3.4 持续学习

RAG 系统是系统持续学习的基础。每天 23:00 运行的 daily-session-vectorize 会将当日产生的所有对话向量化，为系统提供持续增长的学习素材。

学习闭环确保了这个过程不会停止：新的向量会触发质量检查，质量检查会触发优化建议，优化建议会触发新的向量化，形成正向循环。

### 3.5 质疑与进化

Evolution Agent 利用 RAG 系统进行系统级的质疑和改进建议。通过检索历史决策的模式、识别重复出现的问题、分析成功的因素，Evolution 能够提供有数据支撑的质疑和建议。

这种基于事实的质疑比纯粹的理论分析更有说服力，也更容易被系统接受和执行。

## 第四部分：核心價值與未來展望

### 4.1 核心价值

RAG 系统的建立为 AI 治理系统创造了多层次的价值：

第一是知识资产的沉淀。系统不再只是处理当前的请求，而是能够将每次交互的经验转化为可检索的知识资产。这些资产随着时间的推移会持续增值，形成系统独有的智慧积累。

第二是决策质量的提升。通过检索历史经验，系统能够做出更加知情和平衡的决策。决策不再是从零开始的探索，而是有根有据的综合判断。

第三是效率的大幅提升。传统的知识检索往往需要人工翻找，而 RAG 系统能够在毫秒级别返回相关的历史经验，大大压缩了知识获取的时间成本。

第四是系统一致性的保证。通过建立统一的知识库，系统在不同场景下能够保持一致的决策风格和知识应用方式，不会出现前后矛盾的情况。

第五是持续进化的能力。RAG 系统的闭环设计确保系统能够从每次经验中学习，持续提升自己的能力边界。

### 4.2 未来展望

展望未来，RAG 系统还有多个发展方向：

首先是多模态扩展。目前系统主要处理文本向量，未来可以扩展到图像、音频、视频等多模态内容的向量化和管理。

其次是实时学习。目前的向量化是定时批量处理的，未来可以实现真正的实时学习，在对话过程中即时更新向量库。

第三是跨系统联动。未来的 RAG 系统可以与其他 AI 系统共享知识库，形成更大范围的智能协同。

第四是个性化推荐。基于用户的偏好和行为模式，系统可以提供更加个性化的知识推荐，而不仅仅是基于相似度的检索。

第五是自动化优化。引入机器学习算法，让系统能够自动优化向量化的参数、检索的策略、关联的建立，真正实现自我进化的目标。

---

## 結論

經過多日的系統建構，AI 治理系統的 RAG 資料庫已經從一個簡單的知識儲存工具，發展成為一個具有深度理解、廣泛關聯、持續學習能力的智能系統。系統目前的 342 個向量 chunks、456 條 embedding cache、以及完善的三維度質量控制機制，為系統的各個層面提供了強大的知識支援能力。

更重要的是，這個系統不是靜態的。它通過持續同步、質量檢查、三維度優化等閉環機制，確保了知識庫會隨著時間推移而不斷增值、持續優化。這種自我進化的能力，正是 AI 治理系統能夠不斷成長、持續變得更聰明的核心動力。
